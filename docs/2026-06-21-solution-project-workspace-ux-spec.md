# 方案项目工作台 UX Spec

日期：2026-06-21  
状态：v0.2  
适用范围：Deck Master localhost Web UI

## 1. 用户故事

用户进入工作台后，需要快速完成五类任务：

1. 理解当前方案项目处于什么阶段。
2. 逐页判断内容是否成立。
3. 找到缺失依据和质量风险。
4. 推进审批。
5. 判断是否可以交付。

## 2. 核心角色

本轮按单人桌面工作场景设计，角色统一为`方案负责人`。

他在界面里会反复切换三种心态：

1. 审阅心态：这页能不能过。
2. 推进心态：下一步该补什么。
3. 交付心态：现在能不能交付。

## 3. 主流程

### 3.1 打开项目

进入工作台后，用户应立即看到：

- 项目名
- 当前阶段
- 下一步
- 风险摘要
- 当前待处理页面

### 3.2 早期阶段处理

当项目处于`待准备 / 生成中`时：

- 中央显示阶段工作区
- 不强行展示空预览
- 右侧显示项目级阻断和交付链路状态

### 3.3 逐页处理

当项目进入`待审阅`之后：

1. 左侧选择页面
2. 中央看预览
3. 右侧看定位、来源、依据、风险
4. 执行处理动作

### 3.4 补依据

当页面或项目进入`待补依据`：

- 左侧页面项要有明确标记
- 右侧要能看见缺口描述
- 处理动作保留`请求补依据`和`记录备注`

### 3.5 审批

审批对象分两层：

1. 页面级审批
2. 项目级审批

审批任务必须回答：

- 审什么
- 为什么审
- 谁提交
- 等谁判断
- 当前结论

### 3.6 交付

在`可交付 / 已交付`阶段：

- 右上保留`提交审批`和`确认交付`
- 中央支持切到`交付预览`
- 底部推进状态明确显示交付产物是否就绪

## 4. 输入与反馈

### 4.1 用户输入

本轮用户输入只有三类：

1. 选项目
2. 选页面
3. 执行动作并填写备注

### 4.2 系统反馈

系统反馈必须即时刷新：

- 阶段状态
- 页面计数
- 风险计数
- 审批记录
- 活动记录

## 5. 异常流

必须覆盖：

1. 没有项目
2. 项目加载失败
3. 当前没有页面
4. 页面没有预览
5. 交付预览缺失
6. 交付文件存在但不可读

## 6. 完整度标准

当用户完成一次完整闭环时，应能连续完成：

1. 找到当前项目
2. 进入当前页
3. 判断页面
4. 留下处理结果
5. 查看审批状态
6. 切换到交付预览
7. 确认交付或回退处理
