## 2. Office Hour 结论

本轮 Office Hour 已确认以下硬约束。

- 第一用户：售前解决方案架构师，先以当前用户自己的客户 Solution Deck 工作流作为最严苛样本。
- 第一场景：会后客户 Solution Deck 草案，从会议转写、客户材料、历史方案和口头判断出发。
- 10 倍指标：从“方案思路基本确认到第一版可审查草案”的约 12 小时，压到约 2 小时。
- 产品路线：Deck Master 走 `Run OS` 路线，成为专业 Deck 的生产运行时，管理上下文、对话、论点、页面、来源、预览、质量门禁和审批。
- 首版边界：保留人工判断和最终交付前审查；不做实时飞书拉取、不做长期思考库、不做 reference PPT 自动视觉提取。
- 下一轮评审：基于 `Run OS` 路线做 CEO Review，重点审 2 小时目标是否足够硬、会后方案草案是否是最窄可赢切口、Run OS 是否会过早平台化、真实项目能否验证首版价值。

## 3. 当前核心判断

Deck Master 要解决的问题是：如何用 AI 人机协作方式，持续、可靠地产出专业、商用级、可交付的客户 Solution Deck。

这里的 Deck 主要指客户方案、咨询汇报、解决方案架构、业务提案、产品战略汇报等高密度商业表达材料。

这类 Deck 的难点不只在视觉效果。真正的难点在于：

- 把复杂业务判断压缩进有限页面。
- 在每页中同时呈现论点、论证和论据。
- 让页面既能支持演讲，又能作为交付物独立阅读。
- 在历史资料、客户上下文、产品能力和行业判断之间建立清晰关系。
- 找到哪些内容必须保留，哪些内容需要删掉，哪些内容应该放进附录。
- 把高价值思考转化成结构清晰、证据充分、视觉可读的页面。

普通 AI PPT 工具主要解决“不会做 PPT 的人如何快速得到一份看起来不错的 PPT”。Deck Master 关注的是“会做专业 Deck 的人如何把精力从机械成片转回深度思考和客户表达”。

Office Hour 后，Deck Master 的当前阶段目标进一步收敛为：让售前解决方案架构师在客户会议后，用约 2 小时得到一版可审查、可继续打磨的 Solution Deck 草案。

## 4. 用户表达出的真实场景

### 4.1 专业方案写作消耗的是高注意力

当前用户已经具备写高质量方案的能力，问题不在于不会写，而在于这件事高度消耗心力。

一份高质量 Solution Deck 通常需要长时间保持沉浸状态。真正困难的部分包括：

- 想清楚方案主线。
- 决定每页表达什么。
- 判断信息取舍。
- 将复杂内容压缩成有限画布。
- 把观点、证据、案例和视觉结构放到同一页里。
- 反复调整页面密度、层级和表达顺序。

用户希望 Deck Master 把这类工作从一次性高心流写作，转化为可以通过碎片时间和 AI 对话逐步推进的生产过程。

### 4.2 高价值工作发生在对话和判断阶段

用户认为最有价值的工作发生在：

- 和 AI 讨论项目。
- 引入历史会议、客户资料、产品资料和行业研究。
- 共同梳理方案怎么写。
- 抽象出更高价值的业务框架、增长模型、咨询框架和可复用表达。
- 把客户场景背后的共性问题想清楚。

机械成片、找历史页、调结构、改排版、补页面和整理版本等工作，应尽量被系统接管。

### 4.3 Deck 是观点、证据和交付物的组合

专业 Deck 不只是演讲背景，也不只是视觉稿。它通常同时承担三类职责：

- 观点表达：告诉对方核心判断是什么。
- 论证过程：说明为什么这个判断成立。
- 证据呈现：用案例、截图、数据、客户原话、业务模型和产品能力证明判断。

如果页面只有观点，它会退化成大字报。如果页面只有材料，它会变成资料堆叠。专业 Deck 的价值在于把观点、论证和证据压缩成可理解、可讲述、可交付的页面结构。

### 4.4 碎片时间不是闪念工具

用户提到“碎片时间”，核心含义不是要做闪念胶囊或快速记录工具。

更准确的含义是：

- 用户可以通过语音、文字或对话表达阶段性思考。
- AI 负责追问、整理、结构化和补足缺口。
- Deck Master 把这些对话结果转成 Deck brief、claim map、page task 和 preview。
- 用户不必一次性投入完整的大段时间，也能持续推进一份高质量 Deck。

因此 Deck Master 不应建设新的长期思考库。已有的 LLAM wiki、OpenViking、飞书会议记录、项目长期记忆和 Agent 长期记忆继续承担知识库职责。Deck Master 在一次 run 中引用这些上下文，并保存来源指针、摘要、决策和审查记录。

## 5. 产品定位草案

Deck Master 是专业 Deck 创作者的 `Run OS`。

它面向已经理解业务、能够判断内容质量、需要产出商用级 Deck 的专业用户。它的能力重点不放在“自动做一份漂亮 PPT”，重点放在把专业 Deck 的生产过程标准化、可追踪、可审查、可持续迭代。

一句话定位：

> Deck Master helps expert builders turn business thinking, trusted context, and historical deck assets into reviewable, evidence-backed commercial decks.

中文表达：

> Deck Master 帮助售前解决方案架构师和专业顾问，把业务思考、可信上下文和历史优质页面，转化为可审查、证据充分、可交付的客户 Solution Deck。

`Run OS` 的含义：

- 它围绕一次 Deck run 管理状态，避免停留在一次性页面生成。
- 它保存上下文引用、对话判断、论点结构、页面任务、来源决策、质量报告和审批记录。
- 它把用户的精力留给业务判断、论点取舍、证据组织和客户表达。
- 它把资料检索、历史页复用、页面规划、生成交接、预览审查和质量门禁变成可追踪流程。

## 6. 和普通 AI PPT 工具的差异

| 维度 | 普通 AI PPT 工具 | Deck Master |
|---|---|---|
| 目标用户 | 不擅长做 PPT 的普通用户 | 专业顾问、方案架构师、解决方案专家、高质量 Deck 创作者 |
| 主要价值 | 快速生成好看的 PPT | 快速生产可信、可讲、可交付的专业 Deck |
| 核心输入 | 主题、文档、网页、模板 | 用户思考、客户上下文、会议记录、历史方案、知识库、优秀页面 |
| 核心难点 | 排版和视觉生成 | 论点、论证、论据、取舍、证据链、叙事压缩 |
| 产物判断 | 是否好看 | 是否能支撑客户决策和专业表达 |
| 历史资产 | 通常作为模板或素材 | 作为可复用论证结构、案例页、模型页和证据页 |
| 人机关系 | 用户输入主题，AI 生成 PPT | AI 追问、整理、组装、质检，用户做关键判断 |
| 成功标准 | 快速得到一份 PPT | 约 2 小时内得到可审查客户 Solution Deck 草案 |

## 7. 目标用户和第一用户

### 7.1 第一用户

第一用户是售前解决方案架构师，当前用户本人是第一批最严苛样本。

原因：

- 真实场景明确。
- 对方案质量有高要求。
- 有大量历史资料、会议记录、项目上下文和方案库。
- 能快速判断产物是否真的有用。
- 能持续提供反馈，帮助产品形成专业标准。
- 这个角色对客户 Solution Deck 的要求同时包含观点、证据、产品能力和可交付表达。

### 7.2 后续可扩展用户

后续用户可能包括：

- 售前解决方案架构师。
- 咨询顾问。
- 战略/业务分析师。
- ToB 产品和增长负责人。
- 需要持续产出客户方案、投标材料、产品战略汇报的人。

这类用户的共同点是：他们不缺写 PPT 的基本能力，缺的是把高质量思考稳定转成专业 Deck 的系统化生产能力。

## 8. 典型使用场景

### 8.1 会后客户 Solution Deck 草案

用户结束客户会议后，已有会议转写、客户材料、产品资料、历史方案和口头判断，希望在约 2 小时内形成一版可审查的客户 Solution Deck 草案。

Deck Master 应完成：

- 读取本地/已导出的客户资料。
- 引导用户明确受众、目标、核心判断和证据。
- 整理 Deck brief。
- 形成 claim map。
- 检索历史优秀页面。
- 判断每页复用、改写、生成或人工补证据。
- 生成可审查 preview。
- 通过 Draft Gate 检查论点、论证和证据缺口。

成功标准：

- 页面主线可用。
- 证据缺口明确。
- 历史页复用可解释。
- 人工能继续打磨。

失败标准：

- 页面泛泛成稿，像普通 AI PPT。
- 证据链不清。
- 历史资产引用没有理由。
- 用户仍需手工重做主线。

### 8.2 会议后方案整理

用户结束一次客户会议或内部讨论后，已有会议转写和一些口头判断。

Deck Master 应完成：

- 从转写中提取客户问题、关注点、约束和机会。
- 生成一组追问，帮助用户补充关键判断。
- 把会议中的隐性需求转成 Deck 结构。
- 标记哪些论点缺证据，哪些可以从历史方案库复用。

### 8.3 历史方案资产复用

用户已经做过大量优秀页面，但每次新项目仍需手工找、改、拼。

Deck Master 应完成：

- 按当前 claim 和页面角色检索历史页。
- 区分可直接复用、结构可改写、需要新生成和需要人工补证据的页面。
- 保留来源、理由和风险。
- 把用户审批结果反哺后续排序。

### 8.4 高质量框架沉淀

用户在项目中提出了高价值业务模型、增长飞轮、咨询框架或业务判断。

Deck Master 应完成：

- 在本次 run 中识别它们。
- 建议哪些内容值得沉淀到外部知识库或方案库。
- 不在 Deck Master 内建设新的长期笔记库。
- 通过来源指针和审批记录帮助用户后续回收这些高价值资产。

### 8.5 交付前质量门禁

用户已经有一版 Deck 草案，需要判断是否足够专业、可信、可交付。

Deck Master 应完成：

- 检查每页是否有明确主张。
- 检查证据是否支撑主张。
- 检查截图、案例、数据和客户语境是否足够。
- 检查页面密度、重复结构和叙事断裂。
- 进入后续阶段后检查渲染质量、版本一致性和交付包完整性。

## 9. 核心能力模块

### 9.1 Context Intake

负责在一次 run 中引用已有上下文。

首版支持：

- 本地 Markdown。
- 文本文件。
- 会议转写。
- 项目摘要。
- 历史方案说明。
- 已导出的知识库片段。

暂不支持：

- 实时飞书拉取。
- 自动写入 OpenViking。
- 建设 Deck Master 内部长期笔记库。

### 9.2 Guided Deck Conversation

负责把用户思考引导成可编译的 Deck 输入。

应追问：

- 这份 Deck 面向谁。
- 对方看完要做什么决定。
- 核心判断是什么。
- 哪些证据能证明这个判断。
- 哪些历史页面值得复用。
- 哪些内容应该删掉、合并或放进附录。

### 9.3 Deck Brief Compiler

负责把上下文和对话整理成稳定输入。

输出包括：

- 项目名称。
- 目标受众。
- 业务目标。
- 核心观点。
- 必须覆盖主题。
- 风格约束。
- 输出边界。

### 9.4 Claim Map Builder

负责把 Deck 的内容主线转成论点结构。

每个 claim 应包含：

- 论点。
- 重要性。
- 支撑论证。
- 需要的证据。
- 已有证据来源。
- 风险和缺口。

### 9.5 Page Planner

负责把 claim map 转成页面任务。

页面任务应按阶段分层：

- planning：页面职责、核心论点、内容目标、证据需求、视觉需求。
- retrieval：历史页检索词和约束。
- sourcing：复用、改写、生成、人工补证据的决策。
- generation：生成任务说明和状态。

### 9.6 PPT Library Retrieval

负责调用 PPT Library 检索历史优秀页面。

Deck Master 需要拿到：

- slide id。
- 来源文件。
- 页码。
- 截图。
- 文本摘要。
- 置信度。
- 胜率。
- 复用次数。

### 9.7 Sourcing Decision Engine

负责判断每页的来源策略：

- reuse：历史页可直接进入审批。
- adapt：历史页结构可复用，但需要改写语境。
- generate：需要新生成页面。
- manual_placeholder：缺少事实、证据或客户信息，需要人工补充。

### 9.8 Preview Studio

负责让用户审查 Deck 草案。

首版核心操作：

- approve。
- reject。
- note。

后续可扩展：

- 要求替换候选。
- 锁定历史页。
- 转为生成页。
- 标记人工补证据。

### 9.9 Quality Gate

负责质量门禁，是 Deck Master 的内建子系统。

首版覆盖三段 gate：

- Draft Gate：在渲染前检查业务目标、核心论点、页面职责、信息密度、证据链和信息缺口。
- Render Gate：在生成 PPTX 后检查页数、稀疏页、疑似整页截图迁移和渲染后内容退化。
- Delivery Gate：在交付前检查最终 PPTX 页数、禁用词、交付阻断项和资源包完整性。

质量评分维度：

- Narrative Integrity。
- Page Job Clarity。
- Information Density。
- Evidence And Specificity。
- Screenshot And Asset Integration。
- Layout Variety。
- Consulting-Style Expression。
- Visual Readiness。
- Delivery Readiness。

质量结果写入 `quality_reports/`，并在 Web Studio 展示 Pass、Conditional Pass、Rework Required、阻断原因和页面级修复建议。

现有独立 `ppt-quality-gate` skill 保留为可复用能力，但 Deck Master 必须拥有自己的质量运行时、报告和阻断规则。

## 11. 当前主要缺口

### 11.1 引导式对话仍是静态模板

当前只是生成默认追问，并没有真正根据用户回答动态追问。

需要后续补：

- 多轮问答状态。
- 追问策略。
- 用户确认点。
- 已锁定判断和未确认判断区分。

### 11.2 Claim Map 仍是弱抽取

当前 claim map 只是根据上下文摘要和 topic 做基础结构化。

需要后续补：

- 更强的论点抽取。
- claim 去重。
- 论证链构建。
- 证据和 claim 的匹配评分。
- 高价值框架识别。

### 11.3 页面规划仍偏模板化

当前页面结构主要来自内置 page budget 和 role 模板。

需要后续补：

- 工作区页型资产。
- 高密度页面原型库。
- 证据型页面原型。
- 案例拆解页面原型。
- 不同 Deck 类型的叙事结构。

### 11.4 质量门禁仍只覆盖 Draft Gate

当前 Draft Gate 主要检查业务目标、论点和证据缺口。

需要后续补：

- 截图质量检查。
- 页面密度检查。
- 长 Deck 章节质量衰减。
- PPTX 页数和预览页数一致性。
- 交付包完整性。

### 11.5 外部知识源仍未接入

当前首版只处理本地/已导出资料。

需要后续评审：

- OpenViking 查询。
- LLAM wiki 读取。
- 飞书会议记录拉取。
- 语音转写输入。
- 客户敏感资料处理边界。

## 12. 下一轮 CEO Review 问题

在进入下一轮实现前，需要围绕 `Run OS` 路线确认以下问题。

### 12.1 2 小时目标是否足够硬

- 约 12 小时到约 2 小时是否能代表 10 倍价值？
- 2 小时内产出的“可审查草案”最低标准是什么？
- 哪些质量问题可以留给后续人工打磨，哪些问题会让草案失去价值？

### 12.2 会后方案草案是否是最窄可赢切口

- 会后客户 Solution Deck 草案是否比历史页复用、质量门禁更适合作为第一切口？
- 这个切口能否覆盖售前解决方案架构师最高频、最痛的工作？
- 是否需要先锁定一个真实项目作为回归样本？

### 12.3 Run OS 是否会过早平台化

- 当前 `Run OS` 需要保留哪些运行时对象？
- 哪些能力必须进入首版，哪些只进入愿景文档？
- Deck Workspace、资产系统、质量门禁和生成交接的实施顺序是否需要重排？

### 12.4 上下文策略

- 首版是否继续坚持本地/已导出资料优先？
- OpenViking、LLAM wiki、飞书会议记录何时接入？
- 客户敏感资料是否只保存来源指针和摘要？

### 12.5 对话体验

- Deck Master 应该像方案搭档一样主动追问到什么深度？
- 哪些判断必须由用户确认？
- 哪些判断可以让 AI 默认生成草案？

### 12.6 质量标准

- 一份“可审查 Deck 草案”的最低标准是什么？
- Draft Gate 阻断条件有哪些？
- 哪些问题可以进入 preview 后再人工处理？

### 12.7 工程阶段

- 下一阶段优先做对话状态，还是优先做更强 claim map？
- 是否先做 workspace 初始化？
- 是否先把 Preview Studio 变成真正可用的产品入口？

## 13. 建议的后续评审流程

建议按以下顺序推进：

1. Product Review：已通过 Office Hour 初步收敛，确认第一用户、第一场景和核心成功标准。
2. CEO Review：基于 `Run OS` 路线判断方向是否足够尖锐，是否有独特价值。
3. Engineering Review：评审 runtime、数据结构、CLI、UI 和外部工具边界。
4. Design Review：评审 Web Studio 作为专业 Deck 审查界面的信息结构。
5. Implementation Spec：把确认后的方向拆成明确任务包。

## 14. 下一版实现候选

### Option A：Run OS 首版闭环

目标：让一次会后客户 Solution Deck run 从本地上下文、对话、brief、claim map、page tasks、sourcing、preview 到 Draft Gate 形成稳定闭环。

适合解决：

- 12h 到 2h 的核心验证。
- 当前链路仍偏原型。
- 后续 CEO Review 需要真实样本证据。

### Option B：强化 Claim Map

目标：让 `claim_map.json` 真正成为专业 Deck 的内容骨架。

适合解决：

- 论点抽取太弱。
- 证据链不清楚。
- 页面规划偏模板化。

### Option C：强化 Preview Studio

目标：让 Web Studio 从预览工具升级为专业 Deck 审查工作台。

适合解决：

- 当前 UI 产品感弱。
- 用户无法高效判断每页是否成立。
- 质量门禁结果没有足够可视化。

### Option D：Workspace 初始化

目标：让用户能创建标准 Deck 工作区，管理视觉规范、页型资产和质量标准。

适合解决：

- 视觉和页型标准无法持续复用。
- 每次项目都要重新找模板和规范。

## 15. 当前建议

下一轮不要直接继续堆功能。

建议先完成一次基于 `Run OS` 的 CEO Review，确认：

- 2 小时可审查草案是否是足够硬的 10 倍指标。
- 会后客户 Solution Deck 草案是否是最窄可赢切口。
- 首版 `Run OS` 的最小运行时对象。
- Draft Gate 的阻断条件和可放行条件。
- 最小可交付范围。
- 外部知识源接入顺序。

在评审完成前，当前实现可以继续作为原型和验证底座，但不宜直接视为正式产品形态。
