# 方案项目工作台设计评审与截图审计计划

日期：2026-06-21  
状态：v0.2  
适用范围：Deck Master localhost Web UI

## 1. 目标

这份计划用于约束后续设计评审和截图审计，确保工作台不再回退到 Demo 风格或内部语言界面。

## 2. 评审轮次

### Round 1：结构评审

检查：

- 首屏焦点是否清晰
- 项目头部条是否回答了当前项目、当前阶段、下一步
- 左中右分工是否稳定

### Round 2：工作流评审

检查：

- 页面处理是否完整
- 依据链路是否可追
- 风险链路是否可追
- 审批链路是否闭环
- 交付链路是否可见

### Round 3：视觉一致性评审

检查：

- 是否符合 MainQuest `Glass Terminal`
- 是否还有后台拼装感
- 是否还有内部术语残留

## 3. 固定截图场景

1. 无项目空态
2. 待准备
3. 生成中
4. 待审阅
5. 待补依据
6. 待审批
7. 可交付
8. 已交付

## 4. 固定断点

- 1600
- 1440
- 1280

本轮只审桌面，不把移动端纳入主门禁。

## 5. 审计维度

每张图统一审查八项：

1. 首屏焦点
2. 阶段可理解性
3. 页面可见性
4. 操作层级
5. 风险可追溯性
6. 审批闭环
7. 交付预览完整度
8. MainQuest 视觉一致性

## 6. 问题分级

- `P0`：首屏主任务错误、内部术语裸露、交付链路缺失
- `P1`：主次层级混乱、风险追溯不完整、审批信息不完整
- `P2`：视觉细节、留白、按钮一致性、数据排版细节

## 8. 门禁标准

进入下一轮视觉精修前，至少满足：

1. 前台不出现 `run` 和 `audit`
2. `待准备 / 生成中` 不再出现空舞台
3. `待审阅` 之后预览稳定成为主视觉
4. `可交付 / 已交付` 可切到交付预览
5. 页面、风险、审批三条链路完整可达
