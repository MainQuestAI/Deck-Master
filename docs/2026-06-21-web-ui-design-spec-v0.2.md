# Deck Master localhost Web UI Design Spec v0.2

日期：2026-06-21  
状态：实现对齐稿  
适用范围：Deck Master localhost Web UI  
前台产品名：审稿桌

> 2026-06-23 v0.3 覆盖说明：涉及顶部状态带、待准备态安全文案、右侧决策流顺序和前台命名时，以 [Review Desk 修复迭代 Spec v0.3](/Users/dingcheng/Coding-Project/02-key-project/Deck-Master/docs/specs/web-ui/review-desk-remediation-spec-v0.3.md) 为准。`方案项目` 只表示当前处理对象，前台主界面统一收口为 `审稿桌`。

## 1. 结论

本轮 localhost Web UI 已从演示原型转向桌面工作台结构，前台主语统一为`方案项目`。界面不再围绕创建流程和内部运行概念组织，首屏改为围绕五件事展开：

1. 当前方案项目是什么。
2. 目前处于哪个阶段。
3. 下一步该推进什么。
4. 哪些页面需要处理。
5. 当前离交付还有多远。

## 2. 产品定位

方案项目工作台是 Deck Master 在浏览器里的可视化处理界面，负责承接一次方案项目从阶段理解、逐页处理、补依据、推进审批到确认交付的主工作流。

职责边界：

- 负责：阶段判断、页面预览、来源与依据判断、风险处理、审批推进、交付回看。
- 不负责：移动端深度体验、长期知识浏览、多人协同编辑、完整 PPT 编辑。

## 3. 前台语言原则

前台主界面禁止裸露以下内部概念：

- `run`
- `audit`
- 原始状态串
- 技术型命令提示

前台统一术语：

- `run` → `方案项目`
- `workspace` → `方案项目工作台`
- `page review` → `页面处理`
- `delivery preview` → `交付预览`

## 4. 首屏信息架构

桌面首屏采用三层结构：

### 4.1 顶层：项目头部条

固定回答四个问题：

1. 当前方案项目名称
2. 当前阶段
3. 下一步动作
4. 风险摘要与主动作

字段组成：

- 项目名
- 阶段标签
- 更新时间
- 下一步说明
- 风险摘要
- 项目指标
- 项目切换
- 主动作：`提交审批`、`确认交付`

### 4.2 中层：主舞台

主舞台按阶段切换两种模式：

1. `待准备 / 生成中`
   中央显示`阶段工作区`，直接说明当前阻塞、交付预览缺口、责任对象和建议动作。

2. `待审阅 / 待补依据 / 待审批 / 可交付 / 已交付 / 风险冻结`
   中央切回`预览优先`，让页面预览成为首屏主视觉。

### 4.3 侧栏：处理侧栏

右侧固定四段：

1. 页面定位
2. 来源与依据
3. 风险与缺口
4. 审批记录与处理动作

左侧固定为`页面队列栏`，只承担页面顺序、筛选、状态标记和当前页定位，不承接项目切换和创建流程。

## 5. 阶段模型

前台阶段固定为 8 段：

1. `待准备`
2. `生成中`
3. `待审阅`
4. `待补依据`
5. `待审批`
6. `可交付`
7. `已交付`
8. `风险冻结`

每个阶段必须同时给出：

- 一句定义
- 当前阻塞原因
- 下一步动作
- 责任对象
- 预计结果

## 6. 四条主链路

### 6.1 页面处理链路

标准路径：

1. 在左侧页面队列选择页面
2. 在中央查看预览
3. 在右侧确认页面职责、来源和依据
4. 执行`批准页面 / 驳回页面 / 请求补依据 / 升级审批 / 记录备注`

### 6.2 依据链路

每个页面必须能看到：

- 主论点
- 已关联依据
- 缺失依据
- 依据强弱说明

### 6.3 风险链路

每个风险项必须能回答：

- 风险是什么
- 影响哪一页
- 缺什么
- 建议动作是什么
- 当前是否已处理

### 6.4 交付链路

`可交付 / 已交付`阶段必须接入 PPT Master 的交付级产物：

- `render_results/render_result.json`
- `rendered/index.html`

工作台支持两种视图切换：

- `页面处理`
- `交付预览`

## 7. 关键布局要求

### 7.1 左侧页面队列栏

保留：

- 页面筛选
- 页面计数
- 页面状态芯片
- 风险标记

移除：

- 项目列表主视觉
- 创建表单主视觉
- 长段解释文字

### 7.2 中央主舞台

要求：

- 早期阶段不能出现大面积空心预览
- 进入逐页处理后，预览必须成为首屏主视觉
- 交付阶段必须能切到真实交付预览

### 7.3 右侧处理侧栏

要求：

- 页面相关信息优先
- 项目级阻断在早期阶段替代页面明细
- 审批记录和处理动作同区收口

## 8. 空态与失败态

本轮必须覆盖：

1. 无项目空态
2. 项目可打开但还没形成页面
3. 页面无预览
4. 依据为空
5. 审批为空
6. 交付预览缺失
7. 交付文件缺失
8. 当前项目无法加载

空态文案要求：

- 先告诉用户当前处境
- 再告诉用户下一步
- 不暴露底层字段名

## 9. 实现范围

本轮纳入：

1. 桌面工作台首屏重建
2. 前台术语系统清洗
3. 阶段工作区
4. 页面预览优先布局
5. 来源 / 依据 / 风险 / 审批四段处理侧栏
6. 交付预览接入
7. 聚合接口与前端 view-model 改造
8. 桌面截图审计矩阵

本轮不纳入：

1. 移动端专门设计
2. 多人协同编辑
3. 通知中心
4. 复杂权限系统

## 10. 验收标准

### 10.1 设计验收

1. 用户进入首屏 5 秒内能回答当前方案项目、当前阶段和下一步动作。
2. `待准备 / 生成中`阶段中央区必须是有效工作区，不能塌为空白预览。
3. `待审阅`之后中央区必须让页面预览成为主视觉。
4. 任意风险项都能追到页面、依据缺口、建议动作和处理状态。
5. 任意审批任务都能看清原因、等待对象、结论和是否阻塞交付。
6. `可交付 / 已交付`阶段必须能切到交付级预览。
7. 前台不允许出现`run`、`audit`和内部状态串。

### 10.2 实现验收

1. 首屏主渲染只依赖单一聚合接口。
2. 页面动作和项目动作都走统一 action 模型。
3. `render_result.json` 缺失和 `index.html` 缺失都有明确界面表现。
4. 旧 Demo 风格项目列表和首屏空舞台已经清理。
