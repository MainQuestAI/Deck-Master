## Problem Statement

作为 Solution Architect，写商业级 Solution Deck 时，约 **60% 的页面来自历史 PPT 组装**，40% 来自 AI 全新生成。当前流程是：

1. 拿到命题 → PPT Deck Pro Max 共创叙事弧线
2. 生成初版 Deck
3. 人工判断缺口（缺深度、缺架构图、缺案例截图）
4. **翻找 ~500 份历史 PPT，凭记忆定位可复用页面（2-3 小时）**
5. 手动调整复用页的标题、文案以匹配新叙事
6. 拼接组装成终版

核心矛盾：随着 AI 生成内容增多，对单页内容的记忆越来越弱——因为页面不是自己逐字写的。

## Constraints

- 个人开发者，内部提效工具，不做商业化
- Agent-facing：面向 Claude Code / Codex / Hermes 等 AI Agent 提供可调用工具
- 不做复杂前端 UI
- 与 PPT Deck Pro Max 协作（叙事弧线 + 生成），不替代
- 规模：~500 份 PPT → ~50,000 页 → 去重后 ~15,000-20,000 独特页面

### 系统依赖

| 依赖 | 用途 | 必需性 |
|------|------|--------|
| python-pptx | PPTX 文本提取 | V1 必需 |
| numpy | 向量相似度计算 | V1 必需 |
| LibreOffice headless | slide 截图生成 | V1 必需 |
| Embedding API (OpenAI / Infini AI / 豆包) | 语义向量生成 | V1 必需 |
| Pillow | 截图去重哈希计算 | V1 必需 |

## Premises

1. **页级粒度**：复用最小单位是单页 slide，索引和搜索粒度匹配
2. **语义索引**：搜内容语义而非关键词——"资产管理模块的入库到出库流程说明（含 WMS 截图）"
3. **与 Deck Pro Max 协作**：Deck Pro Max 负责叙事弧线+全新生成，PPT Library 负责建库+搜索。组装和 AI 文案适配推迟到 V2
4. **AI 辅助文案适配（V2）**：复用页面标题/文案需 AI 调整以匹配新叙事上下文，人最终审核
5. **Agent-facing CLI 工具**：面向 Agent 提供结构化工具，不搞前端

## Landscape

已调研市面 8 款 Slide Library/Management 产品（DeckFinder、Slidebank、Shufflrr、SlideCamp、Deckmatic、Slideboxx 等）。共同特征：
- 都是 web/SaaS 或 PowerPoint 插件形态
- 关键词/元数据/元素类型搜索
- 面向企业团队（权限、团队文件夹扫描）
- **没有任何产品做"叙事弧线感知"或与 AI 生成流程集成**

关键 Gap：现有工具解决"find a slide"，我们需要"find slides that fill specific narrative gaps in THIS story"。

## Approaches Considered

### 方案 A：CLI 工具集
纯 CLI 命令，Agent 通过 Bash 调用。SQLite + 文件系统存储，嵌入 API 做语义搜索。
- 极简，零外部依赖，立即可用
- 但无结构化工具发现，Agent 需要"记住"命令

### 方案 B：MCP Server
MCP 协议 Server，暴露结构化工具。Agent 自动发现和调用。
- 标准 Agent 工具协议，类型安全，自动发现
- 架构复杂度比 CLI 高

### 方案 C：gstack Skill
包装成 Skill，最快原型验证。
- 最快跑通，利用 gstack 生态
- 不适合长运行服务（索引大量 PPT），最终需迁移

## Recommended Approach: 方案 A — CLI 工具集

**选型理由：** 老板拍板 CLI。最简路径，Agent 直接调 Bash 命令。MVP 只做 `index` + `search`。`assemble` 因跨 PPTX 格式保真度风险推迟至 V2（需先做技术 spike 验证 python-pptx 跨母版 slide 复制可行性）。

### 版本路线

| 版本 | 范围 | 核心价值 |
|------|------|----------|
| **V1 (MVP)** | `index` + `search` + `status` | 把翻找从 2-3 小时压到 5-10 分钟 |
| **V2** | `assemble` + AI 文案适配 | 自动化拼接+文案调整 |
| **V3** | 与 Deck Pro Max 深度集成、`dedup` | 叙事弧线→缺口识别→搜索→组装全链路 |

### 存储模型

```
~/.ppt-library/
  index.db              # SQLite 数据库，~50-100MB
  screenshots/          # 截图，按 SHA256 内容哈希命名，自动去重
    a1b2c3d4....png
    e5f6g7h8....png
```

**SQLite 表结构（V1）：**

```sql
CREATE TABLE presentations (
  id INTEGER PRIMARY KEY,
  path TEXT NOT NULL,           -- 原始 PPTX 绝对路径
  filename TEXT NOT NULL,
  slide_count INTEGER,
  content_hash TEXT,            -- 文件级去重标识
  file_mtime REAL,              -- 文件修改时间，用于增量索引判断
  indexed_at TEXT,
  last_validated_at TEXT        -- 上次验证文件仍存在的时间
);

CREATE TABLE slides (
  id INTEGER PRIMARY KEY,
  presentation_id INTEGER REFERENCES presentations(id),
  slide_index INTEGER,          -- 在原 PPT 中的页码（0-based）
  title TEXT,
  text_content TEXT,            -- 页面所有文本内容
  embedding BLOB,               -- 语义向量（numpy float32 序列化）
  screenshot_hash TEXT,         -- 截图 SHA256 哈希
  metadata_json TEXT,           -- {"chart_types":[],"has_screenshot":true,...}
  UNIQUE(presentation_id, slide_index)
);

CREATE TABLE screenshots (
  hash TEXT PRIMARY KEY,
  file_path TEXT NOT NULL,
  width INTEGER,
  height INTEGER
);
```

### 核心 CLI 命令（V1）

| 命令 | 功能 |
|------|------|
| `ppt-lib index <path>` | 解析单个 PPTX，逐页提取文本+截图+embedding，写入 SQLite。损坏/加密文件跳过并报告 |
| `ppt-lib index --batch <dir>` | 批量索引目录。默认增量模式（按 file_mtime 跳过未变化文件），`--full` 强制全量重建 |
| `ppt-lib search "<query>" --top-k 5 --threshold 0.5` | 语义搜索。输出 JSON（含 slide_id、title、text 摘要、screenshot_path、来源 PPT+页码）。Agent 用 Markdown 表格+截图路径向用户展示 |
| `ppt-lib status` | 库统计：总 PPT 数、总页数、独特截图数、最后索引时间。检测孤立记录（源文件已删除/移动），提示 `--prune`（V2）清理 |

### 搜索实现细节（V1）

向量相似度计算方案：**Python 侧加载全量 embedding 到内存 + numpy 矩阵运算**。

- 20,000 条 × 1536 维（OpenAI text-embedding-3-small）× 4 字节 = **~123 MB**，一台 MacBook 完全可以承载
- 搜索时：query → embedding API → 得到向量 → L2 归一化后 `np.dot(embeddings, query_vec)` 等效余弦相似度 → argsort → top-K
- 不引入 sqlite-vss 等扩展依赖，保持零依赖原则
- V2 如果数据量增长，可考虑切换到 sqlite-vec 或 hnswlib 做近似搜索

### 错误处理协议（V1）

- 损坏的 PPTX：跳过，记录到 `index_errors.log`，继续处理下一个
- 加密/密码保护的 PPTX：跳过，记录警告
- 源文件被移动/删除：`status` 命令列出孤立记录，建议 `--prune` 清理
- 空文本页面（只有图片）：text_content 为空字符串，仍参与截图搜索

### 截图生成策略

- 使用 LibreOffice headless 模式：`soffice --headless --convert-to png --outdir /tmp slide.pdf`
- 单线程每页 ~2-3 秒。全量 500 PPT × 100 页 = 50,000 页，单线程 **30-40 小时**
- **并行策略：** 8 进程 LibreOffice 实例，各分 1/8 文件列表，压缩到 **4-5 小时**
- 首次批量索引建议睡前启动，次日完成。后续增量索引每次只处理新文件

### Embedding 成本

- 以 OpenAI `text-embedding-3-small` 为例：$0.02 / 1M tokens
- 单页平均 ~200 tokens → 50,000 页 = 10M tokens → **~$0.20 总计**
- 速率先不预估——100 RPM 限制，需 ~8 小时。实际建议用 Infini AI 或豆包等国内 API（速率更高、延迟更低）
- **先跑 100 页样本**测算实际 token 数和速率，再定最终选型

### 与 PPT Deck Pro Max 的集成契约

- **输入（来自 Deck Pro Max）：** 叙事弧线识别的内容缺口描述（自然语言文本列表）
- **输出（PPT Library 返回）：** 每个缺口对应的 top-5 候选 slide，含截图路径和元数据
- Agent 负责编排两边的调用顺序，PPT Library 不感知 Deck Pro Max 内部逻辑

## Open Questions

1. **Embedding 模型选型：** 需先跑 100 页样本测算 token 数和成本。候选：OpenAI ada-002、Infini AI embedding、豆包 embedding。选择标准：成本 < ¥5/全量索引、速率 > 200 RPM。
2. **跨 PPTX slide 组装可行性（V2 前置 spike）：** 用 5 份不同来源的 PPTX 测试 python-pptx slide 复制保真度。评估是否需要 LibreOffice UNO API 做 deep copy。此 Spike 必须在 V2 开工前完成。
3. **去重粒度：** V1 只做截图哈希精确去重。V3 评估近似重复检测。
4. **搜索精度验证：** 准备 10 个真实缺口描述，人工标注 ground truth（已知该页面在历史 PPT 中），计算 recall@5、recall@10。
5. **相似度阈值校准：** `--threshold 0.5` 是初始默认值。需要用 ground truth 数据测量 text-embedding-3-small 空间中的余弦相似度分布，确定合理默认值和推荐阈值范围。归入搜索验证 spike。

## Success Criteria

1. **搜索效率：** 从"翻找 2-3 小时"降到"搜索+确认 5-10 分钟"
2. **命中率：** 搜索 recall@10 ≥ 80%（10 个真实缺口中，至少 8 个的 ground truth 页面出现在 top-10 结果中）
3. **库管理：** 支持 500+ PPT、50K+ slide 的索引和搜索
4. **增量索引：** 新增 1 个 PPTX 的索引时间 < 2 分钟

## Reviewer Concerns

评审发现以下问题，已在 V1 设计中处理：

| 问题 | 处理方式 |
|------|----------|
| assemble 跨 PPTX 格式保真风险 | 推迟到 V2，前置技术 spike |
| 向量搜索实现模糊 | 明确为 numpy in-memory 方案，20K×1536 ≈ 123MB |
| 截图生成 30-40h | 8 进程并行压到 4-5h |
| MVP 含 assemble 范围过大 | V1 收窄为 index + search |
| FTS5 表未被使用 | 移除 |
| 无错误处理 | 增加损坏/加密/缺失文件的处理协议 |
| --top-k 默认过大 | 改为 5 |
| `ref_count` 过早优化 | 移除，动态查询 |

## Next Steps

1. **Spike: PPTX 解析可行性** — 用 python-pptx 解析 5 个不同类型的 PPTX，验证文本提取+截图生成的稳定性
2. **Spike: Embedding 选型** — 跑 100 页样本，测 token 数、成本、速率
3. **实现 `ppt-lib index`（单文件）** — 解析→截图→embedding→写 SQLite，每个环节独立验证
4. **实现 `ppt-lib index --batch`（目录）** — 增量模式，8 进程并行截图
5. **实现 `ppt-lib search`** — 查询 embedding → numpy 余弦相似度 → JSON 输出
6. **端到端验证** — 准备 3 个真实缺口描述，跑搜索验证 recall
