## Scope

### Accepted Scope (from Step 0)
7 core modules → 8 core modules (post outside voice), greenfield CLI project:
- `ppt_lib/cli.py` — argparse entry, subcommands: index, search, status
- `ppt_lib/db.py` — SQLite schema (含 index_jobs 表), CRUD, embedding load (WAL mode)
- `ppt_lib/indexer.py` — PPTX parsing, text extraction (fallback), batch indexing (ProcessPoolExecutor x4)

## Architecture Decisions (Post Outside Voice)

### D1: numpy in-memory 全量向量搜索
**Decision:** 方案 A — Python 侧加载全量 embedding 到内存 + numpy 矩阵运算
- 20K × 1536 × 4B = ~123MB, 单机 MacBook 可承载
- L2 归一化后 `np.dot(embeddings, query_vec)` 等效余弦相似度
- 零外部依赖, 不引入 sqlite-vss 等扩展
- V2 数据量增长后可切换 hnswlib/sqlite-vec

### D2: ProcessPoolExecutor 截图并行
**Decision:** 方案 A — Python 进程池, 8 进程 LibreOffice headless
- 全量 500 PPT 从 30-40h 压缩到 4-5h
- 每进程独立 LibreOffice 实例, 避免单实例崩溃影响全量
- `max_workers=8` 可配置

### D3: Embedding Provider ABC 抽象
**Decision:** 方案 A — 抽象基类 + 单实现
- `EmbeddingProvider` ABC 定义 `encode(text: str) -> np.ndarray` 接口
- V1 仅 `OpenAIProvider` 实现 (text-embedding-3-small, 1536 维)
- 后续切换 Infini AI / 豆包 只需新增实现类
- 环境变量 `PPT_LIB_EMBEDDING_PROVIDER` 切换 provider

### D4: argparse CLI 框架
**Decision:** 方案 A — argparse (标准库, 零依赖)
- 3 个子命令: `index`, `search`, `status`
- `index` 支持单文件和 `--batch <dir>` 模式
- `index` 支持 `--full` 强制全量重建
- `search` 支持 `--top-k` 和 `--threshold` 参数
- 所有输出 JSON stdout, 日志/错误 stderr

### D5: 标准测试覆盖
**Decision:** 方案 A — ~25 测试, pytest + pytest-cov
- Test plan artifact: [eng-review-test-plan](dingcheng-unknown-eng-review-test-plan-20260521-191913.md)
- db.py (6), searcher.py (5), embedding.py (4), indexer.py (4), CLI smoke (3), screenshot (2)

### D6: SQLite WAL 模式
**Decision:** 方案 A — `init_db()` 中显式启用 `PRAGMA journal_mode=WAL`
- 允许 index (写入) 和 search (读取) 并发执行
- 一行 PRAGMA, 零成本, SQLite 3.7.0+ 默认支持

### D7: index_jobs 失败恢复表 (Outside Voice)
**Decision:** 方案 A — 新增 `index_jobs` 表追踪每个文件的索引状态
- 字段: `file_path, status (pending/processing/completed/failed), slide_index, error_msg, retry_count, last_error_at`
- `index --batch` 自动跳过 status=completed 的文件 (除非 `--full`)
- 失败文件可单独重试, 中途崩溃可从断点续跑
- 500 个 PPT 跑 4-5h 的场景下, 重跑成本从全量重跑降到只跑失败的

### D8: 多模态 AI 内容理解为主, python-pptx 文本提取为 fallback (Outside Voice)
**Decision:** 方案 A — slide 截图 → 多模态模型 (本地优先: LM Studio/Ollama, 云端 fallback) → 内容描述 → text-embedding
- 本地模型优先: 自动检测 LM Studio / Ollama 是否运行, 检查是否有可用视觉模型 (如 gemma-4, llama-vision)
- 云端 fallback: 用户可配置 OpenAI/Claude API key
- 兜底: 模型不可用时 fallback 到 python-pptx 文本提取
- `metadata_json` 存模型生成的结构化描述 (图表类型、关键结论、适用场景)
- 彻底解决 Codex 提出的文本提取质量、OCR、图表理解、模板噪音问题
- 新增模块: `ppt_lib/vision.py` — 多模态模型调用 + provider 检测

### D9: LibreOffice 进程隔离加固 (Outside Voice)
**Decision:** 方案 A — 每个 worker 独立 user profile + timeout + stderr 捕获
- 每个 LibreOffice 进程使用 `-env:UserInstallation=file:///tmp/lo-{worker-id}`
- 单页截图 timeout 30s, 超时 kill 进程并重试
- stderr 捕获写入 per-file log, 渲染错误可追溯
- `max_workers` 默认降至 4 (8 过高, 内存和稳定性风险)

## Performance Review

### D6: SQLite WAL Mode (已确认)
`PRAGMA journal_mode=WAL` 在 `init_db()` 中启用。允许读写并发。

### Memory Budget
- Embedding 数组: 20K × 1536 × 4B = ~123MB
- SQLite 连接开销: 可忽略
- Python 进程: ~50MB
- **总计: <200MB**, MacBook 16GB+ 完全可承载

### Bottleneck Analysis
- 截图生成: 2-3s/页, 8 进程并行 → 4-5h 全量 (可接受, 一次性批量)
- Embedding API: 100 RPM 限制 → ~8h 全量 (受限于厂商, 非代码层优化)
- 搜索: 123MB 矩阵 × query 向量 → <100ms (单次 dot product)
- SQLite: 简单查询, 50K 行 → <10ms

### No N+1 Issues
- searcher 一次性加载全量, 不做逐行查询
- indexer 逐页写 SQLite (批量 commit)

- Greenfield 项目, 无现有代码可复用
- 设计依赖: python-pptx, numpy, LibreOffice, Pillow — 均为成熟库, 不重复造轮子


### Execution Order
1. Launch A + B + C in parallel (3 worktrees)
2. Merge all. Launch D + E in parallel
3. Merge. Launch F + G in parallel
