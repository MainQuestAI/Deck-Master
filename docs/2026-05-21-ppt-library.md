## Vision

### 10x Check

V1 做的是"搜到历史 slide"。10x 版本是"历史 slide 成为每次写新 Deck 的主动参与者"：

1. **叙事感知搜索** — 不搜"WMS 入库流程"，系统知道你在做"制造业数字化转型方案"，自动浮现："你在 3 个制造业 pitch 里用过这张 WMS 架构图，赢了 2 个。"
2. **自动组装** — 给定叙事弧线和内容缺口，自动拼出完整 Deck，标题已适配新叙事
3. **Slide 战绩追踪** — 每页 slide 记录被用过的 Deal、赢/输结果。搜索排序 = 语义相似度 × 实战胜率
4. **跨 Deck 模式挖掘** — "你在制造业 Deck 的前 5 页几乎总是放 Capability Matrix。这是效果最好的 3 张。"
5. **活体库** — 新 Deck 做完自动入库。库是 compounding asset

### Platonic Ideal

用户说："我需要一个零售客户数字化转型方案。关注全渠道、库存可视化、最后一公里配送。"

系统：分析叙事需求 → 在 500 份历史 Deck 中逐页搜索 → 为每个叙事节拍匹配最佳 slide（不仅语义相似，而且实战验证有效）→ 拼出初版 Deck，标题和过渡文案已适配 → 展示 diff："18 页来自 6 个历史 Deck，7 页 AI 生成，这张架构图在 3 个赢单中出现过。"

30 分钟从零到终版草案。人的工作是 review 和拍板，不是翻文件和贴 slides。

## Accepted Scope (added to V1)

- 项目感知 PPT 扫描 + 版本去重 + 软链接管理 (`discovery.py`)
- 文件系统监听自动增量索引 (`watch.py`)
- 搜索结果 HTML 审查页——左侧文字、右侧截图、轮播翻页、键盘导航、4 状态覆盖 (`html_renderer.py`)
- 本地视觉模型环境诊断 (`diagnostics.py`)
- 搜索结果聚类后处理
- slides 表 source 字段标注内容来源
- JSON 输出 schema 契约
- config.yml 配置文件支持
- 视觉模型优先级链：本地检测 → API key 引导 → python-pptx 兜底
- 测试覆盖更新至 ~40 tests

## Three-Project Architecture

| 层 | 项目 | 回答的问题 |
|----|------|-----------|
| **编排层** | Deck Master | "这个 Deck 怎么做？用哪些历史 + 新生成？" |
| **生成层** | PPT Deck Pro Max | "这一页怎么设计？叙事/视觉/QA？" |
| **数据层** | PPT Library | "历史里有没有类似的？哪张胜率最高？" |

PPT Library 的 metadata_json 作为 Deck Master 未来写入胜率数据的扩展点。

## Codex P1 Findings (Noted, Not Blocking)

- 增量索引语义需明确 rename/move/delete 处理
- SQLite schema 需 slide 级状态字段
- Embedding 限流/重试/缓存需 V1 基础能力
- 搜索冷启动延迟需 .npy 缓存
- Symlink 断链/重名/云盘路径变化需处理
- Diagnostics 需加模型能力白名单和最大图像尺寸检测
- Fallback 质量不一致需 quality_score 或 extraction_warnings
