# Deck Master Web UI 重构审视

日期：2026-06-21
状态：讨论底稿 v0.1
适用范围：`scripts/preview/static/` 当前 localhost Web UI

## 1. 结论

当前 Web UI 已经具备可用雏形，也已经对齐了 Run 审查面板这条产品方向。

现在的主要问题不在“功能太少”，而在“一个界面同时承担了太多层级的表达”：

- 顶部在讲项目级状态。
- 左侧在讲页面级队列。
- 中部在讲视觉预览。
- 右侧在讲审批与证据。
- 底部又回到 run 级推进和论点覆盖。

结果是：

- 用户第一眼能看到很多内容，但很难立刻理解“我此刻应该做什么”。
- 页面主任务和 run 主任务同时抢注意力。
- 视觉气质已经偏高级，但信息组织仍偏工程台账。

这一轮如果要重构，建议把目标定成：

> 把当前界面从“信息很多的项目工作台”，收敛成“围绕一次 run 的高密度审查台”。

## 2. 当前做对了什么

这些部分建议保留，不要推翻：

- 产品边界是对的：Web UI 作为 Agent 唤起式审查台，这个定位清楚。
- 三栏骨架是对的：左侧找页，中间看页，右侧判断，这条主路径成立。
- 顶部状态摘要有价值：阶段、下一步、风险、交付状态都属于高价值信息。
- 页面预览被放在视觉中心，这点正确。
- 页面审批、备注、补证据动作已经落在右侧，符合审查动作路径。
- 已经有空状态和错误兜底，说明产品完成度基础不错。

## 3. 当前主要问题

### 3.1 产品职责表达过满

当前首页同时承载了四个层级：

1. 项目概况
2. 当前 run 状态
3. 单页审查
4. 交付确认

这会让用户在同一屏里反复切换思维层级。

对老板这样的第一用户，真正高频的问题通常只有三个：

- 先看哪一页
- 这页能不能过
- 整个 run 还差什么才能交付

当前 UI 把这三件事和很多“背景解释”放到了同等视觉权重上。

### 3.2 信息架构仍然偏台账

现在界面里的卡片很多，信息块很多，标题也很多，但主次关系不够锋利。

具体表现：

- 顶部四张状态卡都长得很像，用户要逐个读。
- 左侧页卡信息较碎，扫描成本偏高。
- 右侧审查区把“页面定位、来源、依据、风险、审批记录、动作”平铺堆叠，缺少一个更强的判断顺序。
- 底部三个大块再次引入 run 级信息，和右侧形成重复。

### 3.3 视觉语言偏“高级模板感”

当前界面已经避开了很多低级问题，但仍然带有明显的通用 AI 界面指纹：

- `Inter` + 深色玻璃面板 + 白字 + 橙色点缀，是比较常见的默认高级感公式。
- 各个卡片边框、圆角、透明度、阴影非常一致，整体显得安全，但缺少节奏。
- 顶部与下方模块都在同一种玻璃语言里，层次分化不够。
- 中央预览图很大，但外围容器也很大，真正的视觉张力还没有完全释放出来。

### 3.4 交互状态还不够“审查工具化”

当前按钮能用，但“动作后果”和“当前建议动作”表达还不够强。

具体问题：

- 主动作过多：`新建项目`、`提交审批`、`确认交付`、`批准页面`、`驳回页面`、`请求补证据`、`升级审批` 同时出现，焦点分散。
- 当前页如果存在阻断，界面没有把“先处理什么”压到最前。
- 缺少更明确的键盘焦点表现。
- 缺少更成体系的 loading/skeleton 状态，当前以文字空态为主。

## 4. 建议的重构方向

## 4.1 方向一句话

把它做成“审稿桌”，不要做成“后台总控台”。

## 4.2 信息层级调整

建议改成下面这套优先级：

### A 层：当前任务

这是用户打开页面后 5 秒内必须看懂的信息：

- 当前 run 名称
- 当前阶段
- 当前最需要处理的页面
- 当前阻断是否存在
- 当前建议动作

### B 层：页面判断

这是用户做审查决定时必须看的信息：

- 预览图
- 页面职责
- 核心论点
- 来源决策
- 证据充足度
- 风险等级
- 审批动作

### C 层：run 全局补充信息

这些信息有价值，但不该抢首屏权重：

- 论点覆盖
- 历史处理记录
- 交付级预览
- 审批记录列表

建议把 C 层内容收进可展开区域、次级抽屉或切换视图，不要持续占用首屏黄金位置。

## 4.3 版式建议

建议保留三栏结构，但重排权重：

- 左栏更窄，专注“找页”和“状态筛选”。
- 中栏更大，进一步强化预览和页面结论。
- 右栏更像决策台，围绕“能不能过”和“为什么”组织。

底部三块大面板建议降权，优先考虑两种处理方式：

- 合并到右栏下半部分，作为折叠区。
- 做成底部抽屉，只在需要时展开。

## 4.4 视觉语言建议

建议继续使用深色基底，但把气质从“科技玻璃”收敛到“专业审查台”。

建议方向：

- 背景从纯黑感改成带暖灰或墨色偏移的深底。
- 减少整页一致的玻璃面板，改成“少量重点发光 + 更多克制实面”。
- 主强调色保留一个即可，建议沿用偏琥珀/铜橙方向。
- 字体换掉 `Inter`，优先考虑 `Geist`、`Outfit` 或更有识别度的标题字。
- 页码、状态数、置信度等数字信息继续使用等宽数字。
- 标题层级再拉开，当前大标题足够大，但中级标题还不够有统领感。

## 4.5 交互建议

建议把动作结构改成“一个主动作 + 若干次动作”：

- 页面通过时：主动作是“批准页面”。
- 页面不过时：主动作是“驳回并说明”或“请求补依据”。
- `升级审批` 这种动作应在页面已经完成初审后再显著出现。
- `确认交付` 应只在真正到达交付阶段时成为顶部唯一高权重按钮。

同时建议补三类状态：

- skeleton loading：替代“载入中”文字等待。
- 强阻断态：让 P0/P1 风险直接控制页面主按钮状态和说明。
- 完成态：某页已通过后，给出明确的下一页建议，而不是只更新状态。

## 5. 这一轮不建议做的事

- 不建议把 Web UI 做成完整编辑器。
- 不建议引入更多浮层和弹窗。
- 不建议继续堆新的信息卡片。
- 不建议先做大面积动效，当前阶段优先解决结构问题。
- 不建议把“项目管理”心智继续塞进首页。

## 6. 我建议的下一步

如果这轮讨论继续往下走，最稳的顺序是：

1. 先确认 Web UI 的核心身份是否收敛为“审查台”。
2. 再确认首屏到底围绕“当前页”还是“当前 run”展开。
3. 然后我输出一版新的信息架构和低保真布局说明。
4. 结构确认后，再进入真实界面改造和截图审查。
