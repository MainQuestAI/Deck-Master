# Deck Master v1.0.0-rc.1 Release Candidate Evidence

- Status: **Release Candidate (preparation)** — not yet tagged.
- Target: `v1.0.0-rc.1` (after M2 closure stabilizes).
- Companion document: `docs/releases/2026-07-09-1.0.0-iteration-plan.md`.

This document classifies the evidence that Deck Master 1.0.0 readiness rests
on, so external users can tell apart what is **publicly reproducible** from
what is **local-only verification** by maintainers. It deliberately does not
include absolute local paths, private customer names, tokens, or raw customer
material.

## Evidence classification

### Tracked (committed, publicly reproducible from a fresh clone)

- **Benchmark case metadata**: `benchmarks/cases/real_*/benchmark_case.json`
  (3 real-metadata cases: retail growth, healthcare enablement, manufacturing
  GEO expansion) plus `retail_fixture`. Metadata is public; raw source material
  is not (see local-only).
- **Runtime contracts and schemas**: `docs/contracts/`,
  `skills/deck-master/schemas/`, `skills/stage-contracts.json`,
  `product-capability-manifest.json`.
- **Capability and design truth**: `product-capability-manifest.json`,
  `DESIGN.md`, `AGENTS.md`.
- **Release-candidate gate**: `scripts/runtime/rc_gate.py` (two-tier gate),
  `scripts/benchmark/aggregate.py`, and tests under `tests/test_rc_gate*.py`.
- **CI pipeline**: `.github/workflows/ci.yml` (unit tests + the
  `rc_gate_ci` job running `rc-gate --tier ci`).
- **Release tooling**: `release-build` / `release-smoke` / `release-install` /
  `release-rollback` commands and the `release-manifest.json` /
  `deck_capability_lock.json` emitters (relative paths only).

### CI evidence (reproduced on every PR/push)

- Unit tests: `python -m unittest discover -s tests` (Python 3.11 + 3.12).
- Skill OS contract & schema validation (pytest subset).
- Fixture autoplan smoke + Technical Preview gate.
- Release-build smoke.
- **CI-tier rc-gate**: `rc-gate --tier ci` — runs the reproducible subset
  (schema, artifact, release-smoke, fixture-e2e, browser-smoke, capability-lock
  closure) and skips the local-only `benchmark_aggregate` and live
  backend-binding checks. A fresh clone has no `benchmarks/results/` and no
  bound `ppt-master` backend, so the **full** tier would fail by design in CI;
  the CI tier proves the reproducible machinery is green.

### Local-only evidence (maintainer machine, gitignored)

These are real verification artifacts that exist on a maintainer machine but
are **not** committed and **not** reproducible by an external cloner without
the same local state. They are the reason the runtime is "green locally" while
the 1.0.0 gap is "reproducible evidence," not functionality.

- `rc_reports/rc_gate_report.json` — **superseded**. A previous full-tier
  `rc-gate` run with `status: pass` was recorded under an earlier gate version
  (7 checks, no `real_workflow_uat`). The current full tier has **12 required
  checks** including `real_workflow_uat`, `bridge_v2_contract`,
  `sourcing_v2_contract`, `library_status_v2`, and `strict_fixture_policy`.
  The old report does not satisfy the current gate definition and must be
  re-run against the current head (`9bdac61`) after real generation/render/UAT
  evidence is available. Gitignored.
- `benchmarks/results/` — complete report pairs
  (`benchmark_report.json` + `benchmark_rc_report.json`) for the 3 real cases
  plus the aggregate report (`status: report_ready`,
  `complete_real_case_pairs: 3`). Gitignored.
- Real-case source material under a local-only benchmarks directory
  (`raw_source_policy: local_path_only` in each case metadata). Not committed.
- The local `ppt-master` backend binding (a maintainer worktree on a feature
  branch), recorded in the local backend bindings file. `suite-status` reports
  `production_backend_ready: true`; `client_delivery_ready` is **not yet true**
  against the current head — it requires fresh full-tier RC gate, generation
  UAT, and render evidence.

### Boundary (not a blocker, by design)

`agent-doctor --mode production` returns `blocked` only because
`final_readiness` requires a concrete `--run-dir` delivery to validate. Every
other production check (`production_backend`, `capability_lock`, `suite_status`)
passes. This is the "needs a concrete delivery run" boundary, not a functional
gap.

## What closes 1.0.0-rc.1

1. PPT Master clean-environment reproducible bind pinned to a public tag/SHA
   (path documented in `docs/integration/ppt-master.md`); live verification of
   the public repo is out-of-tree.
2. Real benchmark closure archived as **local-only evidence** with a public
   summary (no raw customer material).
3. rc-gate full tier green at release; CI-tier green in CI.
4. Release-tree install/verify/rollback green with evidence.
5. Review Desk browser smoke green (`rc-gate --require-browser-smoke`).
6. `docs/releases/v1.0.0.md` and `docs/user-guide.md` published.

## Out of scope for this RC

- Splitting `scripts/deck_master.py` (separate refactor branch).
- Tagging — tagging `v1.0.0-rc.1` is a separate release decision after the
  above close.
