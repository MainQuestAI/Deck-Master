## 1. Review Verdict

Status: `DONE_WITH_CONCERNS`

这套 spec 的方向成立：Deck Master 需要从单一 skill 升级为 suite runtime，并把 PPT Library、PPT Deck Pro Max、PPT Quality Gate、PPT Master 的独立能力纳入受控编排。

当前不适合直接开工。原因集中在三类阻断：状态检查仍有写入副作用、companion skill 的来源与安装矩阵没有冻结、外部结果契约与现有 Deck Master canonical import 链路存在冲突。

Autoplan 执行方式说明：本环境没有交互式 `AskUserQuestion` 工具，因此本轮采用只读 sequential review，覆盖 CEO / Design / Engineering / DevEx 四个视角，并结合两个并行只读子审查和一轮外部 Codex 只读审查。

## 2. P0 Blocking Findings

### P0-1. `setup-status` 的只读约束和现有实现冲突

Spec 要求 `setup-status` / `suite-status` 必须 non-mutating。Master breakdown 把它列为第一不变式，Stack A 也要求 `suite-status` 和 `setup-status --include-suite` 只读。

现有实现里，`setup_status()` 默认写入 `setup.status.checked` 事件：

- Spec: `docs/deck-master-v0.9.12-stack-spec-pack/deck-master-v0.9.12-master-stack-breakdown.md:35`
- Spec: `docs/deck-master-v0.9.12-stack-spec-pack/deck-master-v0.9.12a-suite-runtime-foundation-spec.md:152`
- Code: `scripts/runtime/setup_status.py:302`
- CLI: `scripts/deck_master.py:878`

额外风险：`setup_status()` 会调用 `validate_skill()`，而 `validate_skill()` 当前也会写安装日志。

- Code: `scripts/runtime/setup_status.py:250`
- Code: `scripts/skills/installer.py:255`

Required spec fix:

- Stack A 增加纯读 inspection contract，例如 `inspect_setup_status()` / `inspect_skill_link()`。
- `setup-status`、`suite-status`、Review Cockpit readiness API 默认走纯读路径。
- 写事件只允许由显式 mutating 命令触发，例如 `setup`、`suite-install`、`suite-repair`。
- 测试必须断言 temporary HOME 下执行 status 后，不新增文件，不改变 mtime。

### P0-2. Suite install / manifest 没有冻结 companion skill 来源矩阵

Master breakdown 承诺安装 Deck Master 后得到完整 suite 或明确缺口报告。Stack A 的 manifest 示例只覆盖 `ppt-library`，没有完整定义 `ppt-deck-pro-max`、`ppt-quality-gate`、可选 `ppt-master` 的来源、版本、能力和阻断规则。

当前仓库只包含 `skills/deck-master/SKILL.md`：

- Spec: `docs/deck-master-v0.9.12-stack-spec-pack/deck-master-v0.9.12-master-stack-breakdown.md:9`
- Spec: `docs/deck-master-v0.9.12-stack-spec-pack/deck-master-v0.9.12a-suite-runtime-foundation-spec.md:37`
- Code: `scripts/skills/installer.py:14`

外部只读 Codex 运行还捕捉到本机 `ppt-library` skill 缺 YAML frontmatter 的预警。这说明 suite status 需要能识别 malformed external skill，并给出修复动作。

Required spec fix:

- Stack A 增加 Companion Skill Source Matrix。
- 每个 companion 写清：
  - required / optional
  - bundled path 或 external adopted path
  - CLI 名称和 tool registry key
  - minimum version
  - required capabilities
  - schema versions
  - Codex / Claude Code link name
  - malformed external skill 的状态枚举和 next action
  - real directory conflict 处理规则
- 如果本轮不打包某个 companion 的 `skills/<name>/SKILL.md`，spec 必须明确它是 external dependency，并定义 adoption validation。

### P0-3. Companion result import contract 和现有 Deck Master canonical contract 冲突

Stack B 同时引入 PPT Library selection、PPT Quality Gate findings、PPT Deck Pro Max generation result 三套新格式，但现有 Deck Master 已有 canonical import 格式和 gate engine。当前 spec 没有把新格式明确设定为 adapter input，容易让实现写出第二套平行链路。

PPT Library:

- Spec 新格式是 `results[]`。
- 现有 reader 读取 `library_results/selection.json` 的 `by_beat`。
- Code: `scripts/tools/ppt_library_client.py:190`
- Code: `scripts/planning/sourcing_decider.py:142`

PPT Quality Gate:

- Spec 新格式是 `deck_master_quality_findings.v1`，severity 使用 `blocking` / `warning`。
- 现有导入格式是 `deck_external_quality_review.v1`，severity 使用 `P0/P1/P2`，交付阻断依赖 `blocks_delivery`。
- Code: `scripts/quality/external_review.py:27`
- Code: `scripts/quality/external_review.py:138`
- Code: `scripts/quality/external_review.py:247`

PPT Deck Pro Max:

- Spec 新格式是 `ppt_deck_pro_max_generation_result.v1`，status 使用 `result_submitted`，outputs 放在嵌套对象里。
- 现有导入格式是 `deck_generation_result.v1`，status 使用 `completed/partial/failed`，路径字段在顶层，且必须是 run-relative。
- Code: `scripts/generation/handback.py:29`
- Code: `scripts/generation/session.py:456`
- Code: `scripts/generation/session.py:495`

Required spec fix:

- Stack B 明确：companion output 是外部输入，Deck Master 内部仍以现有 canonical artifacts 为准。
- 每条 import 命令必须先 validate adapter input，再转换为现有 canonical format。
- bad JSON / unsupported schema / mismatched run_id / mismatched session_id 必须 rejected，且不得覆盖旧 artifact。
- import 后统一写 `imports/import_log.jsonl`。

## 3. P1 Required Fixes Before Implementation

### P1-1. CLI 名称和参数需要和现有入口对齐

Stack A 要求 `suite-status`、`suite-install`、`suite-repair`、`setup-status --include-suite --output json`、`install-skill --suite`。当前 parser 没有这些 suite 参数或命令。

- Spec: `docs/deck-master-v0.9.12-stack-spec-pack/deck-master-v0.9.12a-suite-runtime-foundation-spec.md:142`
- Code: `scripts/deck_master.py:1457`
- Code: `scripts/deck_master.py:1672`

Required spec fix:

- 明确 `deck-master` wrapper 是 public command，测试必须覆盖 wrapper。
- `--output json` 作为兼容参数，默认 JSON 行为保持不变。
- `install-skill --suite` 只能扩展现有 single-skill install，不能引入第二套安装系统。

### P1-2. PPT Library selection 需要精确兼容写入规则

Stack B 要求 import 后同时写 canonical 路径和 legacy 路径，但还缺字段映射细则。

- Spec: `docs/deck-master-v0.9.12-stack-spec-pack/deck-master-v0.9.12b-companion-workflow-contracts-spec.md:118`
- Spec: `docs/deck-master-v0.9.12-stack-spec-pack/deck-master-v0.9.12b-companion-workflow-contracts-spec.md:156`
- Code: `scripts/tools/ppt_library_client.py:181`
- Code: `scripts/planning/sourcing_decider.py:148`

Required spec fix:

- 增加 `results[] -> by_beat` 映射表。
- `page_task_id`、`slot_id`、`query_trace_id` 必须保留到 selected candidate 或 decision metadata。
- 多个相同 role 的 beats 必须用 `beat_id/page_task_id/slot_id` 区分。

### P1-3. Generation session 状态流需要冻结

Stack B 测试要求 result import 先停在 `results_imported`，preview refresh 后再进入后续状态。现有 `generation-session import-results` 会尝试立即 refresh preview，并在成功时置为 `quality_required`。

- Spec: `docs/deck-master-v0.9.12-stack-spec-pack/deck-master-v0.9.12b-companion-workflow-contracts-spec.md:385`
- Code: `scripts/generation/session.py:508`

Required spec fix:

- 二选一冻结：
  - 保持现有自动 refresh，测试改为期望 `quality_required`。
  - 改造为两步流程：import 到 `results_imported`，显式 refresh 到 `preview_refreshed/quality_required`。
- 如果保留 `session_id`，import 必须校验当前 run 的 active generation session。

### P1-4. Quality findings 必须作为 adapter 复用现有 gate engine

Stack B 的 `gate_class` 和 `blocking/warning` 能作为外部输入，但 Deck Master 最终 gate artifact 必须继续输出现有 `deck_quality_report.v1`。

- Spec: `docs/deck-master-v0.9.12-stack-spec-pack/deck-master-v0.9.12b-companion-workflow-contracts-spec.md:191`
- Code: `scripts/quality/external_review.py:260`

Required spec fix:

- `import-quality-findings` 定义为 thin adapter。
- `gate_class -> scope/gate file` 固定映射。
- `blocking -> P1` 或按规则升级 `P0`，`warning -> P2`。
- `title` / `description` 统一映射到现有 `message`。
- `summary.delivery_ready` 只能作为输入提示，最终 delivery readiness 由 Deck Master 重新计算。

### P1-5. Import log 应该前置到 Stack B

Stack B 已要求 companion outputs 写 import log，Stack C 才定义 import log shape，顺序有冲突。

- Spec: `docs/deck-master-v0.9.12-stack-spec-pack/deck-master-v0.9.12c-production-closure-governance-spec.md:115`
- Code: `scripts/quality/external_review.py:291`

Required spec fix:

### P1-6. Feedback event CLI 缺少定位字段

Stack C 的 `record-library-feedback --run-id --outcome` 不足以定位哪一页、哪个 candidate、哪次选择。

## 5. CEO Review

Product direction: pass with concerns.

这轮规格对准了真实用户痛点：Deck Master 被点名后要先 setup、再编排、再回收结果，不能让 Agent 直接掉到 PPT Deck Pro Max 或 Canvas 式生产。

主要产品风险：

- 用户安装后仍看不到完整 suite 的可用状态。
- companion skill 可独立使用，但进入 Deck Master run 时没有强制 handback。
- Review Cockpit 如果只剩 JSON fallback，用户会感觉 Deck Master 仍没有把链路串起来。

Product recommendation:

- Stack A 先解决安装可解释性和 first-run ceremony。
- Stack B 只做 adapter 和 import，不扩 companion 深业务。
- Stack C 必须给 Review Cockpit 至少一个最小状态面板，避免发布后仍像黑盒。

## 6. Design Review

Design risk: moderate.

本轮视觉设计范围很轻，但 Review Cockpit 是用户确认链路是否被 Deck Master 接管的关键界面。当前 spec 的 UI 约束还偏弱。

Required design acceptance:

- 顶部 readiness banner：setup / suite / workspace / production readiness。
- Run detail 里显示 imported library selection、generation session、quality findings、feedback pending。
- Blocking quality finding 必须在页面上可见，不能只存在 JSON。

## 7. Engineering Review

Engineering risk: high until P0 is fixed.

核心工程原则应该是：外部 companion 输出都只是输入，Deck Master 内部状态源继续使用现有 canonical artifact。当前 spec 对这个原则写得不够硬，导致最容易出现第二套 selection、第二套 quality gate、第二套 generation result。

Implementation order recommendation:

1. Fix pure-read status foundation.
2. Freeze companion source matrix.
3. Add shared import log.
4. Add library adapter and legacy writer.
5. Add quality findings adapter.
6. Add generation result adapter and state decision.
7. Extend Review Cockpit readiness output.
8. Add release smoke.

## 8. DevEx Review

Developer experience risk: high.

当前最大的 DevEx 问题是第一步命令没有形成稳定的公共表面：

- 文档写 `deck-master setup-status --include-suite --output json`。
- 现有 parser 没有 `--include-suite` / `--output`。
- status 命令当前会写状态事件。
- suite install 的输入来源仍不明确。

Target first-run experience:

1. Agent 读取 Deck Master skill。
2. Agent 执行纯读 readiness check。
3. 若缺 workspace，Agent 向用户询问 workspace。
4. 用户确认后 Agent 运行 setup。
5. Agent 验证 suite readiness。
6. Agent 给出短 readiness summary 和下一步动作。

## 11. Final Gate

Autoplan gate: `DONE_WITH_CONCERNS`

当前 spec pack 可以作为方向稿继续迭代，但应先修 P0 和 P1，再进入实现。若直接进入开发，最可能出现的返工是：suite 安装链路重写、quality gate 结果无法阻断交付、PPT Deck Pro Max handback 无法导入现有 run state、Review Cockpit 看不到真实 suite 状态。

建议下一步：直接在这套 spec pack 上修订 P0/P1，并把 adapter-first 原则写成 v0.9.12 的全局不变式。
