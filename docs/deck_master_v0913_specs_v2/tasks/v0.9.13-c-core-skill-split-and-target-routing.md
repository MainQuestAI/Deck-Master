# v0.9.13-C — Core Skill Split and Target Routing

## Goal

Split agent-facing Core Skills without splitting runtime code. Validate both Codex and Claude Code target routing.

## Core skills

```text
deck-master
deck-planner
deck-review
```

Optional later:

```text
deck-learning
```

## Skill responsibilities

### deck-master

Router, setup, workspace, run-state, suite readiness, handoff guard.

First checks must start with:

```bash
deck-master setup-status --include-suite --output json
```

The Skill must explain missing readiness in natural language, infer or confirm workspace, run safe setup / repair / migration actions, then verify status again. It must not simply hand a raw setup command to the user.

### deck-planner

Context intake, brief, claim map, narrative plan, page task, sourcing intent.

If setup or required suite readiness is blocked, `deck-planner` must return to `deck-master` setup / run-state instead of creating production artifacts.

### deck-review

Quality gates, Review Cockpit, findings, repair loop, export readiness, delivery readiness.

`deck-review` must read Deck Master `run-state` before delivery judgment and must not mark delivery-ready while render / quality / workspace gates are blocked.

## Required target parity

Both targets must be tested:

```text
codex
claude-code
```

## Acceptance

- Core skills exist in release tree.
- Codex symlinks exist and point to release tree.
- Claude Code symlinks exist and point to release tree.
- No real directory is silently overwritten.
- `deck-master` routes to `deck-planner` and `deck-review` in playbooks.
- `deck-master` Skill includes Agent-driven setup ceremony.
- `deck-planner` and `deck-review` docs preserve Deck Master run-state ownership.
- Product Capability Skill docs state active-run output must return through Deck Master import / state update.
- No runtime code duplication.

## Tests

- Skill package files exist.
- Codex fresh install symlink smoke.
- Claude Code fresh install symlink smoke.
- Existing real dir -> migration required, not overwrite.
- Skill text does not instruct production users to bypass Deck Master run state.
- Skill text does not tell users to manually run raw setup commands without Agent guidance.
