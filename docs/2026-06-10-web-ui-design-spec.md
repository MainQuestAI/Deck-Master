# Deck Master Web UI Design Spec

日期：2026-06-10  
状态：确认版 v0.1  
适用范围：Deck Master Run OS MVP 的 localhost Web UI

## 1. 设计定位

Deck Master Web UI 是任务唤起式 Run 审查面板。

它服务一次具体 Deck run，帮助用户在浏览器中完成页面预览、来源审查、质量风险判断、页面级操作、审批和导出前确认。

Agent 仍是主操作入口。用户在 Agent 主会话中完成需求澄清、方案讨论、run 调度、错误解释和下一步推进。Web UI 只承载 Agent 对话界面不适合承载的可视化审查工作。

首版不做持续操作工作台，不做完整项目管理，不做长期知识管理，不做移动端主路径。

## 2. 核心用户与场景

第一用户：售前解决方案架构师、专业顾问、高质量 Solution Deck 创作者。

第一场景：会后客户 Solution Deck 草案审查。

典型输入包括会议转写、客户资料、历史方案摘要、口头判断、workspace 中的视觉规范和质量标准。

典型输出包括 deck brief、claim map、page tasks、sourcing decisions、preview manifest、quality reports 和可导出审批队列。

## 3. Agent 与 Web UI 分工

Agent 负责：

- 需求澄清。
- 方案讨论。
- run 创建和调度。
- 工具调用解释。
- 错误恢复建议。
- 下一步动作建议。

Web UI 负责：

- 页面预览。
- 页面列表扫描。
- 来源决策审查。
- 候选页查看。
- 证据状态查看。
- 质量风险查看。
- 页面级操作。
- 审批和备注。
- 导出前状态确认。

所有 Web UI 操作必须写回 run 状态，让 Agent 可以继续接手。

## 4. 首屏行为

首次打开 run 时，直接进入第一页。

同一 run 再次打开时，恢复用户上次查看页面。

不设置独立总览页。总览信息融入顶部状态条、左侧页面列表和右侧当前页审查面板。

用户可以通过左侧筛选快速跳到阻断页、待审页、生成页和已批准页。

## 5. 整体布局

首版采用桌面三栏布局。

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ 顶部状态条                                                                    │
│ Workspace | Run | 阶段 | 下一步 | Draft Gate | Build Skill | 导出 | 语言       │
├───────────────┬──────────────────────────────────────────────┬───────────────┤
│ 左侧导航       │ 中央预览                                      │ 右侧审查       │
│               │                                              │               │
│ Run 切换       │ 页码 / 章节 / 页面标题                         │ 页面职责       │
│ 页面筛选       │                                              │ 核心论点       │
│ 页面大纲       │ 当前页预览图                                  │ 来源决策       │
│ 状态计数       │                                              │ 证据状态       │
│               │ 上一页 / 下一页 / 全屏查看                       │ 质量发现       │
│               │                                              │ 操作 / 审批    │
├───────────────┴──────────────────────────────────────────────┴───────────────┤
│ 底部状态抽屉：事件日志 | Build Skill 步骤 | Artifact handback | Export queue     │
└──────────────────────────────────────────────────────────────────────────────┘
```

整体信息密度按专业审查界面设计：预览大、信息紧凑、状态清晰。

页面预览保持中心视觉权重。来源、证据、质量风险和审批信息保持直接可见。低频细节可以折叠，关键风险不能折叠到用户难以发现的位置。

## 6. 顶部状态条

顶部状态条用于让用户在 5 秒内判断 run 是否可继续。

信息顺序：

1. Workspace 名称。
2. Run 标题和 run id。
3. 当前阶段：`intake / planning / sourcing / building / draft_gate / review / export_ready`。
4. 下一步动作。
5. Draft Gate 状态。
6. Build Skill 状态。
7. 导出状态。
8. 语言切换。

设计规则：

- 状态必须短，不写长说明。
- 阻断状态必须比普通状态更显眼。
- 点击状态可以打开底部状态抽屉。
- P0/P1 会影响顶部状态。

## 7. 左侧导航

左侧导航用于让用户快速找到需要处理的页面。

模块：

- Run 切换：显示最近 run。
- 页面筛选：全部、阻断、待审查、已批准、已拒绝、生成页。
- 页面大纲：页码、页面标题、页面角色、来源类型、审批状态、质量标记。
- 状态计数：阻断数、待审数、已批准数、生成数。

页面列表默认按 Deck 页码顺序排序，保留 PPT 阅读节奏。

阻断页、待审页、生成页通过筛选和标记处理，不改变默认顺序。

页面项示例：

```text
03  关键挑战
problem · reuse · 待审查 · 1 个质量风险
```

设计规则：

- 左侧不显示大段理由。
- 阻断页、失败生成页、缺资产页要有明显标记。
- 30 到 60 页时仍能扫描。
- 默认排序不能因为筛选以外的状态自动跳动。

## 8. 中央预览

中央预览用于判断页面视觉和内容是否可审查。

模块：

- 当前页标题。
- 页码和总页数。
- 章节上下文。
- 页面预览图。
- 上一页 / 下一页。
- 全屏查看。

设计规则：

- 预览图是视觉中心。
- 缺图时显示可恢复错误，不显示空白框。
- 页面加载中要有明确 loading 状态。
- 全屏查看只服务视觉判断，不承载审批主操作。

## 9. 右侧审查面板

右侧审查面板用于判断当前页是否成立。

信息顺序：

1. 页面职责：这页承担什么角色。
2. 核心论点：这一页要证明什么。
3. 证据需求：需要什么证据。
4. 来源决策：`reuse / adapt / generate / manual_placeholder`。
5. 决策理由。
6. 候选来源：历史 PPT、页码、置信度、截图状态。
7. 质量发现：严重级别、问题、修复建议。
8. 页面操作：替换来源 / 转生成页 / 锁定历史页。
9. 审批：待审查 / 已批准 / 已拒绝。
10. 备注。

审批区示例：

```text
审批状态
( ) 待审查
( ) 已批准
( ) 已拒绝

页面操作
[替换来源] [转生成页] [锁定历史页]

备注
[ 多行输入框 ]

[保存审批]
```

设计规则：

- 批准、拒绝、备注是页面审查主路径。
- 页面操作是二级操作，默认可见，但视觉权重低于审批按钮。
- 替换来源只能从当前页已有候选中选择。
- 转生成页必须显示生成任务是否已创建。
- 锁定历史页必须显示锁定状态和来源页。
- 页面操作必须显示后果说明。
- 所有操作都要写回 run 状态，并进入事件日志。

## 10. 质量风险呈现

质量风险按严重级别分层。

- P0：强阻断。
- P1：强提示。
- P2：轻提示。
- P3：轻提示或数量提示。

P0/P1 必须影响顶部状态、左侧页面列表和右侧审查面板。

P2/P3 默认显示数量和简短说明。

用户尝试批准 P0/P1 页面时，必须显式确认。

底部日志只作为补充证据，不作为主要风险入口。

## 11. 底部状态抽屉

底部状态抽屉用于高级用户和 Agent 操作追踪。

默认收起。

Tabs：

- 事件日志。
- Build Skill 步骤。
- Artifact handback。
- Export queue。

设计规则：

- 默认不打扰普通审查。
- 失败时自动提示可展开。
- 只显示和当前 run 相关的信息。
- 不能把关键阻断只放在抽屉里。

## 12. 双语规则

首版实现中文和英文两套完整语言包。

默认语言跟随浏览器。用户显式切换后，写入本地偏好，优先级高于浏览器语言。

同一屏不能出现中英文混排。

必须进入语言包的内容：

- 按钮。
- 标题。
- 导航。
- 空状态。
- 错误提示。
- 操作说明。
- 审批文案。
- 质量风险说明。

可以保留英文代码形态的内容：

- 文件名。
- 状态枚举。
- 命令。
- artifact id。
- run id。
- source id。

## 13. 视觉方向

Web UI 视觉方向：专业咨询审查台。

关键词：

- 高对比。
- 克制。
- 清晰。
- 商务。
- 生产工具感。

页面预览是视觉中心，状态和审查信息服务判断效率。

避免创意工作室式装饰，也避免开发者控制台式日志优先。

## 14. 桌面范围

首版只保证桌面浏览器体验。

主要设计目标屏幕：13 到 16 英寸笔记本。

移动端不进入本轮验收范围。

CSS 可以避免明显破版，但不承诺移动端审查体验。

## 15. 状态覆盖

| 功能 | 加载中 | 空状态 | 错误 | 成功 | 部分完成 |
|---|---|---|---|---|---|
| Workspace | 正在读取工作区 | 提示创建或选择 Workspace | 显示路径和修复建议 | 显示 Workspace 名称 | 标记缺失配置文件 |
| Run 列表 | loading | 提示从 Agent 或 brief 创建 run | 显示 runs 目录读取失败 | 显示最近 run | 标记 pending run |
| 页面列表 | loading | 提示还没有 preview manifest | 显示 manifest 错误 | 显示页面 | 标记缺图、阻断、待生成 |
| 页面预览 | loading | 提示选择页面 | 显示缺资产路径和恢复建议 | 显示预览图 | 显示占位图和生成状态 |
| Build Skill | 运行中步骤 | 未开始 | 显示失败步骤和日志摘要 | 显示产物路径 | 显示部分页面完成 |
| Draft Gate | 正在检查 | 未运行 | 显示检查失败原因 | 显示通过 | 显示条件通过和修复建议 |
| 审批 | 保存中 | 待审查 | 保存失败可重试 | 保存成功 | 页面已审但 run 未完成 |
| 导出 | 正在生成 | 无批准页 | 导出失败 | 显示 approved_queue | 部分页可导出 |

## 16. 实现交接要求

下一轮实现前需要补齐：

- `DESIGN.md`
- `docs/design-tokens.json`
- `docs/design-tokens.css`
- 中文语言包。
- 英文语言包。

设计系统至少包含：

- 颜色 token。
- 字体 token。
- 状态 token：success / warning / danger / blocked / pending。
- 严重级别 token：P0 / P1 / P2 / P3。
- 间距和圆角 token。
- 页面预览画布规则。
- 审批状态颜色规则。
- 质量严重级别颜色规则。
- 双语文案 key 命名规则。

## 17. 验收标准

- 同一屏不能出现中英文混排。
- 中文和英文两套语言包必须完整覆盖用户可见文案。
- 用户 5 秒内能判断 run 是否可继续。
- 用户 30 秒内能找到所有阻断页。
- 用户能在右侧面板理解当前页的来源、证据和质量风险。
- 首次打开 run 直接进入第一页。
- 同一 run 重新打开时恢复上次查看页面。
- 页面列表默认按 Deck 页码顺序排序。
- 审批状态只使用 `needs_review / approved / rejected`。
- 页面操作首版支持替换来源、转生成页、锁定历史页。
- 页面操作不能绕过 Draft Gate 阻断。
- 页面操作必须可撤销或可由 Agent 在下一步解释和恢复。
- Build Skill 失败必须可见。
- Draft Gate 阻断必须可见。
- 缺图、坏 manifest、生成失败都不能只显示技术错误。
- 首版验收只覆盖桌面浏览器。

## 18. 非目标

首版不做：

- 本地桌面 App。
- 完整 Web 工作台。
- 长期知识管理。
- 实时飞书拉取。
- 移动端主路径。
- 多 Build Skill 并行调度界面。
- 高保真品牌视觉稿。
